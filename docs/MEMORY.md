# Blueprint — Implementation Memory

Status: living log, updated at the end of every completed phase milestone (per `RULES.md` §20) — not a design document, not a duplicate of `DECISIONS.md`. This file answers "where did implementation actually land, and why, when it differed from the plan" — a chronological record for whoever (human or agent) picks this project back up after time away.

"Underway" (per `README_ARCHITECTURE.md` §8) means: implementation begins the moment the first line of application code is written, not when a phase fully ships. Update this file incrementally as milestones inside a phase complete, not only at phase boundaries.

Each entry: date, what shipped, what deviated from `PHASES.md`/`ARCHITECTURE.md` and why (if anything), what's next.

## 2026-07-13 — Project initialized, Phase 0 underway

Planning documents (`PRD.md`, `ARCHITECTURE.md`, `RULES.md`, `PHASES.md`, `DECISIONS.md`, `README_ARCHITECTURE.md`) moved from conversation context into `docs/` as the durable source of truth. Repository initialized with git, monorepo scaffolding per `ARCHITECTURE.md` §18 stood up.

Two tooling ADRs added during scaffolding, not anticipated by the original doc set: **ADR-016** (uv, pinned to CPython 3.12, for the Python side — the 3.14 system default lacks mature native-extension wheel coverage for Tree-sitter/LangGraph/psycopg) and **ADR-017** (npm workspaces, not pnpm/Turborepo, for the JS monorepo — pnpm isn't installed on this machine and Phase 0 has no build graph complex enough to justify Turborepo). Both are explicitly conditional, revisit-on-signal decisions, not permanent commitments.

PR1 (repo scaffolding + CI) shipped: `apps/api` (FastAPI app factory, RQ worker entrypoint, Pydantic settings, Phase 0 SQLAlchemy models), `apps/web` (Next.js via create-next-app), `packages/ui` and `packages/shared-types` (placeholders — nothing to export until Phase 1's `FindingCard` and a real typed API surface exist), `infra/docker` (compose file for local Postgres+pgvector and Redis, a shared Dockerfile for the api/worker services), and CI (lint/typecheck/test for both apps). Verified locally: ruff, mypy --strict, pytest, eslint, tsc, and `next build` all pass.

PR2 (DB schema + migrations) shipped: Alembic wired to `apps/api`'s own settings (one `DATABASE_URL`, not a value duplicated into `alembic.ini`); one hand-authored migration (`f7826c23e482_phase_0_schema.py`) creating exactly the 8 Phase 0 tables — chosen over `alembic revision --autogenerate` because autogenerate requires a live DB connection to diff against, which wasn't available (see below).

**Two documentation findings, filed rather than silently resolved, per the instruction that doc/code disagreement is a bug to file:**

1. `ARCHITECTURE.md` §3.1 describes Stage 1 as extracting commit metadata (author, timestamp, message, files touched), and §11 lists a `commits` table (and separately an `issues` table) as part of the schema — but `PHASES.md`'s Phase 0 deliverable list names only 8 tables, omitting both. Resolved by *not* creating `commits`/`issues` in this migration: nothing in Phase 0's stated acceptance criteria (a populated Repository Graph, zero LLM calls) requires commit or issue history, and the first real consumers (the Debt Agent's staleness signal, Phase 3; feature/dependency reasoning that might reference issues) are both later-phase. Revisit when a shipping stage actually needs either table — don't build storage ahead of a consumer.
2. `ARCHITECTURE.md` §10 leaves the embedding model choice open pending Stage 4's own accuracy/cost comparison, but pgvector needs a fixed column width today. Resolved with **ADR-018**: a documented, provisional 1536-dim placeholder, the same "ship a flagged placeholder, revisit on real data" pattern ADR-009 already established for the absence-claim confidence ceiling — not a new precedent, a reapplication of one already accepted.

**Environment constraint, noted rather than papered over:** this development environment has neither Docker nor a local PostgreSQL installation, so the migration's `upgrade()`/`downgrade()` could not be executed against a real `pgvector`-enabled Postgres here. What *was* verified: `alembic upgrade head --sql` and `alembic downgrade <rev>:base --sql` (Alembic's offline mode, which compiles the DDL through SQLAlchemy's real Postgres dialect without a live connection) — both rendered correct, dependency-ordered SQL, including the `VECTOR(1536)` column type and the `USING hnsw` indexes. CI (`.github/workflows/ci.yml`) runs `alembic upgrade head` against a real `pgvector/pgvector:pg16` service container on every push, so the first PR/push against this repo is the actual live-execution verification — flagging this explicitly rather than asserting untested code works. If you have Docker locally, `docker compose -f infra/docker/docker-compose.yml up -d && cd apps/api && uv run alembic upgrade head` would confirm it directly before that first push.

PR4 (Tree-sitter extraction) shipped, out of the PHASES.md suggested order (ahead of PR3) — deliberately: PR3 (GitHub App auth) needs a real GitHub App registered in GitHub's UI, a manual step outside this environment's reach, while PR4 has no external dependency at all. `pipeline/ingestion/facts.py` defines the Stage 1 output as pure Pydantic DTOs (`SourceFileFacts`, `FunctionFact`, `ClassFact`, `ImportFact`, `TodoFact`) with zero ORM/DB coupling, consistent with `pipeline/` staying importable standalone (ARCHITECTURE.md §13). `discovery.py` walks a repository excluding vendored/generated directories and lockfiles, classifying every recognized source extension — not just the three Tree-sitter-backed languages. `pipeline/ingestion/extractors/{python,typescript_js,go}.py` do real Tree-sitter-grammar extraction (functions, classes/methods with nesting, imports, parameter names/types, return types) plus TODO-to-enclosing-symbol resolution; `heuristic_extractor.py` is the regex-based fallback ARCHITECTURE.md §4 specifies for languages recognized but not yet grammar-backed (Java, Kotlin, Ruby, Rust, C/C++, C#, PHP, Swift), always tagged `StructuralConfidence.LOW` and honest about what it can't determine (no types, no enclosing-symbol resolution for TODOs). `extract.py` is the single top-level entrypoint (`extract_file`, `extract_repository`) that decides Tree-sitter-vs-heuristic per file and guarantees deterministic (sorted) output, which the content-hash caching design in ARCHITECTURE.md §9 will depend on once it's wired up in v1.1.

One design correction made while building this, not present in the original docs: `ARCHITECTURE.md` §4 is easy to misread as "discover only the three initially-supported languages" — but its own failure-mode paragraph requires the opposite (unsupported languages get *discovered and heuristically extracted*, not skipped). `discovery.py` was written, then corrected, to separate "is this file source code at all" (a broad, versioned extension list) from "does it get a full Tree-sitter parse" (the narrow three-language `SUPPORTED_LANGUAGES` set) — catching this before it shipped rather than after.

Tested against a small hand-built fixture repo (`tests/pipeline/ingestion/fixtures/sample_repo/`) covering Python, TSX, Go, a heuristic-fallback Java file, an excluded `node_modules/` file, and an excluded lockfile — 8 new tests, all passing, plus the full existing suite (9 total), ruff, and mypy --strict all clean. This is real, executable verification (unlike PR2's migration, which could only be verified offline) — Tree-sitter and its language grammars are pure Python-extension dependencies with no external service to be missing here.

PR5/6 (Knowledge Graph + Repository Graph construction, ARCHITECTURE.md §3.2–3.3) shipped together, built specifically to make the v1.1 incremental-indexing work (§9) a straightforward extension later rather than a rearchitecture — an explicit request, not something the original doc set called for by name, though §9 already commits to "the schema should support it from day one." Three concrete decisions carry that intent:

1. **DECISIONS.md ADR-019**, a new migration (`51198ced1801`) adding a nullable `file_id` FK to both `graph_nodes` and `graph_edges` — populated for Knowledge Graph rows (1:1 with a file), null for Repository Graph rows (roll up many). This is the indexed answer to "which graph rows does a changed file own," which incremental invalidation has to answer somehow; without it, that question is either a full-graph rebuild or an unindexed `metadata` jsonb scan. `ARCHITECTURE.md` §11 updated to document the new columns rather than left stale.
2. `pipeline/graph/knowledge.py`'s `build_file_nodes(facts: SourceFileFacts) -> list[NodeSpec]` is a pure function of exactly one file's Stage 1 output — proven, not just asserted, by a test that builds nodes for one file standalone vs. as part of a full-repository pass and asserts byte-for-byte equality. Import-edge resolution (which genuinely needs the whole repo's file set) is a separate, explicit second pass (`build_import_edges`) that only touches each file's path and its already-extracted `ImportFact`s, not its parsed structure — so a future "only file X changed" recompute reruns pass 1 for X alone and pass 2 using the unchanged files' cheap paths, not their full facts.
3. `pipeline/graph/repository.py`'s Stage 3 rollup groups files by nearest manifest-bearing ancestor directory (`package.json`/`pyproject.toml`/`go.mod`/`Cargo.toml`/`Dockerfile`/`setup.py` — `ARCHITECTURE.md` §3.3's own "conventional services/packages layout" signal, applied literally rather than via a fixed folder-depth guess) computed per file, so a future incremental rollup for one changed module doesn't need the whole Knowledge Graph, just the files belonging to that module.

Both `pipeline/graph/` modules are pure — no database, no filesystem I/O (the one filesystem scan, `find_manifest_directories`, lives in `pipeline/ingestion/discovery.py` and is passed in as a plain `frozenset[str]`, keeping the rollup itself as testable as the Knowledge Graph builder). `NodeSpec`/`EdgeSpec` (`pipeline/graph/specs.py`) are label-keyed, not DB-ID-keyed, deliberately — the shape a future content-hash cache would memoize per file, and what keeps `pipeline/graph/` from touching a database at all. `services/ingestion_service.py` and `services/graph_service.py` do the actual label → DB-ID resolution and persistence, one snapshot at a time.

Deliberately out of scope, documented rather than silently dropped: "calls" and "references" edges (`ARCHITECTURE.md` §3.2 names all three as Knowledge Graph edge types). Resolving calls needs call-expression facts Stage 1 doesn't currently extract (`FunctionFact` captures signatures, not bodies) — a Stage 1 extension and a follow-up PR, not something skipped by oversight. Import-edge resolution itself is also honestly partial: Python (dotted-path suffix matching) and TypeScript/JavaScript (relative `./`/`../` specifiers only, no tsconfig `paths` alias resolution) are implemented; Go is not (package import paths need `go.mod`'s module name to map to local files, not implemented) — every one of these boundaries is asserted by a test (`test_build_import_edges_skips_unresolvable_and_external_imports`) rather than left to silently produce wrong or missing edges without anyone noticing.

Tested against the same fixture repo PR4 built, extended with two files specifically to exercise import resolution with genuine in-repo targets (`importer.py`, a dotted `import utils.helper`; `web/app.tsx`, a relative `./component` import) and three manifest files (`utils/pyproject.toml`, `web/package.json`, `service/go.mod`) to exercise manifest-driven module/service boundaries against a real cross-module dependency edge. 9 new unit tests, all passing, no database required. Also added a real Postgres integration test (`tests/services/test_persistence_integration.py`) exercising `persist_files`/`persist_graph` end to end — it skips gracefully here (no Docker/Postgres, same constraint as PR2) via a `db_session` fixture that checks connectivity first, but will execute for real in CI against the same `pgvector/pgvector:pg16` service container the migration check already uses.

Full suite: 18 passed, 1 skipped (the integration test, as designed), ruff and mypy --strict both clean.
